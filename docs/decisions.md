# 需求取捨決策

> 這份文件由 Codex 先產生草稿，最後必須由學員刪除、修改或補充。不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：

- 暫定優先服務資訊整理者，因為目前 Phase 0 工作台已經聚焦在原始資訊、草稿判斷與人工確認風險。
- 這個選擇不是定案，仍需要小隊討論與真人確認。

## 這個使用者最想完成什麼

- 看懂每筆原始資訊目前能不能被相信，以及為什麼不能直接用。
- 分清楚原文、人工判斷、AI 或系統預填，避免下一位協作者誤把草稿當事實。

## 這個使用者最怕什麼錯誤

- 模糊或未確認資訊被整理成看起來很正式，導致後續直接行動。
- AI 補了原文沒有的地點、需求、狀態或安全判斷，但畫面沒有清楚標示。

## 我今天暫時不處理的需求

- 暫時不做真正的回報表單流程。
- 暫時不做行動者可執行任務清單。
- 暫時不新增後端、資料庫、外部 API、地圖或 runtime LLM。

## 我不採用的 AI 建議

- AI 建議：
  - 把自動預填草稿視為可直接產生候選通報。
  - 不採用原因：
    - 自動預填只能當整理起點；未確認內容不能變成已確認資料，也不能直接變成志工任務。
- AI 建議：
  - 用「信心程度高」代表比較接近已確認。
  - 不採用原因：
    - 信心程度不是查核狀態；`needs_review` / `unverified` 不得顯示成 confirmed / verified。

## 必須保留人工確認的判斷

- 資訊是否已確認。
- 轉述者是否為第一手來源。
- 地點、時間、需求與安全狀態是否足以行動。
- 是否可以公開涉及個資、健康、長者、地址或親友狀況的資訊。
- 是否能把候選整理轉成任務。

## 今天明確不做什麼

- 不修改 `src/`，因為 Release 01 是需求分析，不是實作階段。
- 不查外部社群、地圖、政府資料或新聞補內容。
- 不把 persona sub-agent 模擬結果當成實際訪談結論。

## 我仍不確定的地方

- 自動預填應該保留並強化標示，還是先拿掉以降低誤解風險。
- 行動者在 v1 是否應該看到草稿，或只看到經人工確認的內容。
- 回報者需要的低門檻輸入與整理者需要的完整判斷紀錄，應該如何分流。
