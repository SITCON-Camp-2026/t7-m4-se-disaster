# Main Agent 訪談彙整

> 這份文件由 Codex 依三個 persona sub-agent 回饋整理。以下是需求分析草稿，不是產品決策；請保留人類確認痕跡，並在小隊討論後補充。

## 三個 persona 的共同需求

### 使用者明確說的

- 都需要清楚分辨「未確認」「需要人工確認」「不能直接行動」。
- 都需要知道資訊取得方式、查核狀態、更新時間，以及目前內容是不是整理草稿。
- 都擔心 AI 或系統自動預填把不確定內容變得像已確認事實。
- 追加的三位資料管理者都要求保留原文、來源、草稿狀態、人工判斷依據，避免交接時只剩狀態標籤。

### 我推論的

- v1 若繼續使用「自動預填」，需要更明確標示哪些欄位來自原文、哪些是人工判斷、哪些是系統推測。
- 「下一步」這個詞可能太像行動指令；不同角色會把它理解成不同風險。
- 資料管理者可以接受系統提示，但會把提示當成檢查提醒，不會把可信度高中低當成查核結果。

### 仍需要人類確認的

- 真人回報者是否真的看得懂「候選類型」「信心程度」。
- 真人行動者看到「候選通報」是否會誤以為可以行動。
- 真人資料管理者是否能清楚分辨「資料可信度高」「人工確認為真實」「準備下決策」三者差異。

## 三個 persona 之間的衝突需求

### 使用者明確說的

- 回報者希望不完整資訊也能先送出；資訊整理者和行動者希望資訊不完整時不要被拿去行動。
- 回報者不想被逼著判斷類型；資訊整理者需要足夠欄位留下判斷脈絡。

### 我推論的

- 同一筆原始資訊可能需要分成「回報時先保留」和「整理後才能行動」兩種視角。
- 快速輸入和安全判斷之間不能只靠更多欄位解決，因為欄位越多可能越讓回報者放棄。

### 仍需要人類確認的

- v1 是否優先改善回報流程，或優先改善資訊整理流程。
- 若只能選一個主要使用者，是否先服務資訊整理者來降低誤判風險。

## Phase 0 原型中被推翻的假設

- 原本假設：
  - 只要標示「自動預填只是草稿」，使用者就能理解它不是已確認內容。
  - 被什麼回饋推翻：
    - 三個 persona 都提到自動預填可能被誤解；行動者尤其擔心「下一步」看起來像可以照做。
- 原本假設：
  - 「信心程度」可以幫助整理者理解草稿品質。
  - 被什麼回饋推翻：
    - 資訊整理者指出「信心程度高」可能被下一位協作者看成「已確認」。
- 原本假設：
  - 只要把「人工確認為真實」放進人工可調欄位，就能避免和系統判斷混淆。
  - 被什麼回饋推翻：
    - 追加資料管理者訪談指出，若「可信度高」和「人工確認為真實」出現在同一組選項或同一視覺層級，值班整理時仍可能誤讀。
- 原本假設：
  - 「準備下決策」可以表示交給人繼續處理，不代表派工。
  - 被什麼回饋推翻：
    - 三位資料管理者都覺得這個詞接近行動語氣，需要再確認是否會讓下一位協作者誤以為可以執行。

## 最容易造成誤導的風險

- 把 AI 自動預填、人工草稿、原文內容混在一起顯示，讓使用者看不出來源。
- 把「下一步」寫得像行動指令，導致行動者誤把候選整理當成已確認任務。
- 只顯示更新時間但沒有說明時間風險，讓過期資訊看起來仍可使用。
- 把「採用為未確認整理草稿」在交接時簡化成「已採用」，讓未確認草稿被下一位協作者當成可執行資料。
- 只改狀態但沒有留下誰確認、根據什麼確認，會讓「人工確認為真實」失去可追溯性。

## 可以考慮的設計方向

> 這裡只是可考慮方向，不是標準答案。

- 把欄位來源標示清楚，例如：原文、人工判斷、系統預填。
- 把「下一步」改成更保守的語氣，例如「建議交由誰確認」或「整理建議」，避免像派工指令。
- 對回報者保留低門檻輸入，但在整理與行動視角明確擋住未確認內容。
- 把「可信度」與「查核結論」分成不同概念與位置，不讓「可信度高」靠近「人工確認為真實」。
- 在狀態變成「人工確認為真實」或類似高風險結論時，要求留下人工依據；但這是需求方向，不代表現在要修改程式。

## 需要人類決策的取捨

- 取捨 1：
  - 選項 A：v1 優先讓回報者快速送出不完整資訊。
  - 選項 B：v1 優先讓資訊整理者分辨原文、人工判斷與 AI 預填。
  - 我目前傾向：選項 B。
  - 理由：目前 repo 已有 Phase 0 原始資訊與整理工作台，下一步若先改善整理者視角，比較能降低把未確認資訊變成行動的風險。
- 取捨 2：
  - 選項 A：保留自動預填，強化標示。
  - 選項 B：先移除自動預填，只留下人工整理。
  - 我目前傾向：暫時保留為可討論問題，不在 Release 01 定案。
  - 理由：三個 persona 都指出風險，但它也可能幫助學員看見 AI 推測風險；需要真人試用後再決定。
- 取捨 3：
  - 選項 A：讓資料管理者先快速分流，每筆只留下最小理由。
  - 選項 B：要求每筆都先寫完整判斷理由，才允許改狀態。
  - 我目前傾向：先列為人類決策問題。
  - 理由：大量資料分流需要速度，但交接與查核風險需要理由；三位資料管理者沒有給出同一個答案。
- 取捨 4：
  - 選項 A：保留「準備下決策」語氣。
  - 選項 B：改成更明確的「交給人工決策前整理」或其他較不像行動指令的文字。
  - 我目前傾向：不在 Release 01 直接改 UI，但把它列為 v1 文字風險。
  - 理由：這次是需求分析階段，不修改 `src/`；但三位資料管理者都指出這是高風險誤解。

## 不應該讓 AI 自動決定的事情

- 不應讓 AI 決定資訊是否已確認。
- 不應讓 AI 決定能不能派人、去哪裡、做什麼。
- 不應讓 AI 補真實地址、真實人物、電話、官方狀態或救災判斷。
- 不應讓 AI 把轉述資訊改寫成第一手確認。

## 我採用的 AI 整理

- 採用三個 persona 都指出的共同風險：自動預填、查核狀態、草稿判斷容易被混淆。
- 採用「資訊整理者」作為 v1 優先服務對象的暫定方向，因為它和目前 Phase 0 工作台最接近，也最能處理未確認資料風險。
- 採用追加資料管理者訪談的共同觀察：「可信度高」「人工確認為真實」「準備下決策」是需要人類再檢查的狀態語氣風險。

## 我不採用的 AI 建議

- 建議：
  - 直接把 persona 回饋轉成 v1 功能清單。
  - 不採用原因：
    - Release 01 是需求分析，不是實作階段；persona 回饋是模擬訪談，不應直接變成產品規格。
- 建議：
  - 把「信心程度高」當成接近可用。
  - 不採用原因：
    - 信心程度不是查核狀態，未確認資訊不能因此變成可行動資訊。
- 建議：
  - 只靠狀態標籤完成交接。
  - 不採用原因：
    - 資料管理者需要看到原文、系統提示、人工判斷依據與仍不確定處；狀態標籤不足以支撐交接。

## 我自己補充的判斷

- 目前最需要處理的是「文字語氣造成的誤解」：候選、下一步、信心程度、自動預填都可能讓草稿看起來太正式。
- v1 應該避免把回報者、整理者、行動者的需求塞進同一個未區分的流程。
- 追加訪談讓我更確定：即使只服務資料管理者，系統也不能只追求分流效率，還要保留能被下一位協作者追溯的人工理由。

## 我仍不確定的地方

- 真人回報者是否願意在危急或混亂情境下填更多欄位。
- 真人資訊整理者是否需要更細的狀態分類，或只需要更清楚的標示。
- 行動者是否應該在 v1 看到候選草稿，還是只看到已經人工確認過的整理結果。
- 資料管理者在大量資料情境下，應該先快速暫分，還是每筆都先補完整判斷理由。
- `docs/phase0-observations.md` 應該補到什麼程度，才足以和畫面草稿、`docs/ai-log.md` 一起作為交接依據。
